Why sl-dbg

Debuggers were built for humans staring at an IDE. AI agents need something else.

Modern coding agents can read code and write patches, but until now they could not actually run a debugger. sl-dbg fixes that. The same tool that ships JSON to an LLM also drops a clean breakpoint workflow into your terminal.

⚡

Stateless wire protocol

No REPL, no session juggling. Each command is one short round-trip the agent (or you) can retry, log, and reason about.

{ }

JSON everywhere

Structured input, structured output. Easy to script, easy for an LLM to consume, never a screen-scrape.

⇄

Built on DAP

Reuses the same Debug Adapter Protocol VS Code uses, so it speaks to mature, well-tested adapters out of the box.

★

MCP-native

Every command is exposed as a Model Context Protocol tool. Drop it into Claude Desktop, Cursor, or Continue in one line.

⌘

Daemon-backed

A tiny local daemon keeps debug sessions alive across invocations, so successive commands stay fast and stateful where it matters.

∅

Zero footprint

Single static binary. No Python or JVM on the host; language adapters install on demand.

Install

One curl. No PAT. No ceremony.

Source lives at y0geshpatil/sl-dbg (Apache-2.0); prebuilt release binaries live in the mirror y0geshpatil/sl-dbg-releases, so installation needs no GitHub login.

install.sh — recommended
$ curl -fsSL https://y0geshpatil.github.io/sl-dbg-site/install.sh | bash

Auto-detects macOS/Linux × amd64/arm64. Falls back to sudo only if the target directory is not writable.

install.sh — pin v0.1.0
$ curl -fsSL https://y0geshpatil.github.io/sl-dbg-site/install.sh | bash -s -- v0.1.0

Useful for CI pinning and reproducible environments.

install.sh — no sudo
$ INSTALL_DIR=$HOME/.local/bin \
    curl -fsSL https://y0geshpatil.github.io/sl-dbg-site/install.sh | bash

Make sure $HOME/.local/bin is on your PATH.

manual tarball
# 1. Grab the tarball for your OS/arch from:
https://github.com/y0geshpatil/sl-dbg-releases/releases/latest

# 2. Extract and put on PATH:
$ tar -xzf sl-dbg_*_darwin_arm64.tar.gz
$ sudo mv sl-dbg /usr/local/bin/

No checksum hassle — every release ships a SHA-256 file you can verify before extracting.

remove everything we added
# Removes the binary AND the sl-dbg entry from every detected agent's MCP config.
$ curl -fsSL https://y0geshpatil.github.io/sl-dbg-site/uninstall.sh | bash

# Only the binary, keep MCP entries:
$ curl -fsSL https://y0geshpatil.github.io/sl-dbg-site/uninstall.sh | bash -s -- --keep-mcp

# Or, surgically, just one agent:
$ sl-dbg mcp uninstall claude   # or cursor | vscode | codex | copilot | all
$ sl-dbg mcp uninstall all --dry-run   # preview first

The uninstaller removes only the sl-dbg server entry — every sibling MCP server in your config is preserved. A timestamped .bak of each modified file is kept next to the original.

Install language adapters

No adapter is bundled. They live in their own ecosystems and you only install the ones you need.

Python
sl-dbg install-adapter python

Wraps debugpy. Installed via pip install --user.

Go
sl-dbg install-adapter go

Wraps dlv dap. Installed via go install.

Java
sl-dbg install-adapter java

Bundled DAP launcher. Needs an existing JDK 11+ on PATH.

All
sl-dbg install-adapter all

One-shot — installs every adapter above.

Quickstart

From zero to a paused breakpoint in two commands.

Debug a Python script
python — local launch
$ sl-dbg start --lang python --program ./app.py --stop-on-entry
{"ok":true,"session":"a1b2","state":"paused","location":{"file":"app.py","line":1}}
$ sl-dbg break app.py:42
$ sl-dbg continue
$ sl-dbg locals
{"variables":[{"name":"user","value":"User(id=7, email='...')","type":"User"}, ...]}
$ sl-dbg eval "user.email"
Attach to a running JVM
java — attach to JDWP
$ sl-dbg attach --lang java --host 127.0.0.1 --port 5005
$ sl-dbg break com.example.OrderService:128
$ sl-dbg threads
$ sl-dbg stack
Debug Go with Delve
go — package debug
$ sl-dbg start --lang go --program ./cmd/server --stop-on-entry
$ sl-dbg break-fn main.handleRequest
$ sl-dbg continue

For AI agents

Drop sl-dbg into Claude Desktop, Cursor, or any MCP client.

sl-dbg speaks Model Context Protocol over stdio. Every CLI command is also an MCP tool — so the same operations work from a terminal and from an agent loop, with identical semantics.

1 · Register with one command

install for your agent
$ sl-dbg mcp install claude   # Claude Desktop
$ sl-dbg mcp install cursor   # Cursor
$ sl-dbg mcp install vscode   # .vscode/mcp.json in cwd
$ sl-dbg mcp install codex    # Codex CLI (~/.codex/config.toml)
$ sl-dbg mcp install copilot  # GitHub Copilot CLI
$ sl-dbg mcp install all      # every detected agent

Safe read–merge–write with a timestamped .bak backup. Pass --dry-run to preview, --force to overwrite, or --print to dump the snippet without touching files.

2 · Sandbox untrusted agents

safe defaults
$ sl-dbg mcp --read-only
$ sl-dbg mcp \
    --allow-cwd /repo \
    --deny-program rm

--read-only hides every mutating tool. --allow-cwd / --deny-program gate debug_start and debug_attach with POLICY_DENIED before they reach the daemon.

What an agent sees

  • Tools — every CLI command (start, attach, break, continue, locals, eval, watch, …) with a strict input schema.
  • Resources — sl-dbg://sessions, sl-dbg://events, sl-dbg://adapters for live state without spending a tool call.
  • Prompts — turnkey templates like diagnose_loop_bug, trace_call_path, watch_then_continue.

Reference

The full command surface, at a glance.

Lifecycle

  • start — launch a program under the debugger
  • attach — attach to a running process or DAP server
  • stop — terminate a session
  • restart — restart the current session
  • sessions — list active sessions

Breakpoints

  • break — set line breakpoint (file:line or Class:line)
  • break-fn — set function breakpoint
  • breaks — list breakpoints + hit counts
  • unbreak — remove breakpoint(s) by id
  • --if, --hit, --log, --once modifiers

Stepping

  • continue — resume execution
  • step — step over
  • step-in · step-out
  • until file:line — run until a line
  • pause — break a running program

Inspection

  • stack · threads
  • locals · globals
  • eval · set
  • print — pretty-printed deep inspect
  • fields — drill into a variable ref
  • snapshot — full pause snapshot

Watches & events

  • watch add · watch eval · watch list
  • events — recent DAP events
  • output — captured stdout/stderr from debuggee
  • Watches are persisted across sessions for the same program.

Meta

  • adapters — list installed adapters
  • install-adapter <lang|all>
  • daemon stop|status|logs
  • version
  • mcp — speak MCP over stdio

Every command returns JSON. Run any of them with --help for the full flag set.

FAQ

Honest answers.

Is the source code open?

Yes. Source lives at github.com/y0geshpatil/sl-dbg under Apache-2.0. The release tarball is identical to what you'd get from go build on the source, with -s -w ldflags to strip symbols.

Do I need a GitHub account to install?

No. Releases live in the public mirror y0geshpatil/sl-dbg-releases, so curl | bash works anonymously.

How do I do a security review?

Everything is auditable: source at github.com/y0geshpatil/sl-dbg, signed release binaries with SHA-256 checksums, the full MCP tool surface, the threat model, and the install scripts. Please report vulnerabilities privately per SECURITY.md rather than filing public issues.

What state lives on disk?

Just a small daemon log, watch-expression cache, and ring-buffered event logs under $XDG_STATE_HOME/sl-dbg (defaults to ~/.local/state/sl-dbg). Safe to delete any time.

What if my SESSION_NOT_FOUND errors come and go?

The daemon respawned — sl-dbg now returns a dedicated DAEMON_RESPAWNED code with a hint to re-launch the session. Common after a host sleep or reboot.

How do I uninstall?

One-liner: curl -fsSL https://y0geshpatil.github.io/sl-dbg-site/uninstall.sh | bash. That removes the binary and the sl-dbg entry from every detected agent's MCP config (with a .bak of each modified file). To skip the MCP cleanup, append -s -- --keep-mcp. To target a single agent: sl-dbg mcp uninstall claude (or cursor / vscode / codex / copilot / all). Cached state lives under ~/.local/state/sl-dbg and is safe to rm -rf.

Can I run multiple sessions at once?

Yes. Sessions get short IDs; pass --sess <id> on any command to target a specific one. Most commands also support a default session so you usually don't need to.
