# Security review & source access

The `sl-dbg` source repository is private, but **we make the code
available to anyone with a legitimate security-review need** — including
enterprise security teams, AppSec auditors, journalists, researchers,
and customers performing pre-adoption due diligence.

This page tells you what is public today, what you can request, and how
to request it.

---

## 1 · What's already public

You can verify a great deal *without* the source:

| Artifact | Where | What it lets you check |
|---|---|---|
| **Release binaries** | [github.com/y0geshpatil/sl-dbg-releases/releases](https://github.com/y0geshpatil/sl-dbg-releases/releases) | Run, fuzz, strace, syscall-trace the actual shipping artifact. |
| **Per-release SHA-256 checksums** | `sl-dbg_<version>_checksums.txt` on each release | Tamper-evidence: confirm the binary you ran is the binary we built. |
| **Install scripts** | [install.sh](install.sh) · [uninstall.sh](uninstall.sh) | Read every byte that runs as part of `curl \| bash`. |
| **MCP tool surface** | [mcp.md](mcp.md) | Auto-generated from a live binary — exact JSON Schema of every input the binary accepts from an AI agent. |
| **Threat model** | [security.md](security.md) | What the daemon trusts, what it doesn't, every policy check, every error code. |
| **Configuration & env vars** | [configuration.md](configuration.md) | Every knob that changes runtime behavior. |
| **Architecture** | [design.md](design.md) | Daemon ↔ adapter ↔ MCP layering and why. |

> 🔍 **For most security questions, the above is enough.** Threat model
> + JSON Schemas + binary you can fuzz often answers "what can this
> reach on my machine?" without needing source.

## 2 · What you can request

If your review requires source-level inspection, you can request one of:

### a · Read-only collaborator access (preferred)

A free GitHub account with **read access** to the private
`y0geshpatil/sl-dbg` repository for the duration of your review (default
90 days, renewable). You see the full git history, can clone, can browse
PRs, can run `gosec` / `govulncheck` / `semgrep` / `trivy` locally.

**You'll need to:**

1. Send an email to **security@sl-dbg.dev** (or open a private issue at
   `y0geshpatil/sl-dbg-releases`) with:
   - Your GitHub username
   - One-line description of why (e.g. "AppSec review for adoption at Acme Corp")
   - Your affiliation
2. Counter-sign a one-page reciprocal NDA (we provide it; it permits
   publication of *findings*, restricts publication of *source*).

We typically respond within **2 business days**.

### b · Source tarball with NDA

If you can't accept a GitHub invite (air-gapped review env, etc.), we
can send a signed `.tar.gz` of the source at a tagged version, hashed
against the corresponding release.

### c · Aggregated reports we already have

Available immediately, no NDA:

- Latest `govulncheck` run output
- Latest `gosec ./...` output
- SBOM in CycloneDX 1.5 JSON format (generated by GoReleaser; one per
  release, attached to the GitHub release page as
  `sl-dbg_<version>_sbom.cdx.json` from v0.4+)

Email **security@sl-dbg.dev** and we'll send the current report.

## 3 · Responsible disclosure

Found something concerning? Please **do not open a public issue**.

- Email **security@sl-dbg.dev** with steps to reproduce.
- We acknowledge within **24 hours** and aim to ship a fix within
  **14 days** for high-severity issues (RCE, sandbox escape, MCP policy
  bypass).
- We credit reporters in the release notes unless asked otherwise.
- We do not yet operate a bug-bounty program but will retroactively
  award goodwill to anyone who reports a verified high-severity issue.

---

## 4 · Why isn't the source public?

The codebase is in active stabilization and the API surface — both CLI
and MCP — is still allowed to break between minor versions. Going
source-public locks in patterns we may want to revise. The plan is to
flip the source repository to **public** once we ship `v1.0.0`; until
then this private-source / public-binary / open-review model lets us
move fast without sacrificing your ability to audit.

---

## 5 · What we'll never do

- Ship telemetry — `sl-dbg` makes **zero network calls** at runtime
  unless you invoke `debug_attach <host>:<port>` yourself.
- Auto-update — the binary you installed is the binary you keep running
  until you re-run `install.sh` or `brew upgrade`.
- Read or transmit your source code — the daemon caches a small ring
  buffer of recent DAP events locally under `~/.local/state/sl-dbg/`,
  nothing more.

You can confirm all three with `strace` / `dtrace` / `Little Snitch`
against a real release binary; we'd rather you didn't have to take our
word for it.
